모든 것을 Gemini Enterprise Agent Platform 네이티브로 — 최대 채택 계획

플랫폼 감사에서 드러난 자체구현(8) · 스텁(7) · 부재(7) 컴포넌트를 전부 관리형 플랫폼 제품으로 전환합니다. 각 항목은 공식문서 · 공식 예제 · 학습 자료 레퍼런스와 단계별 계획을 답니다.

기준: 공식 Google Cloud / ADK / A2A / AP2 문서 + 샘플 리포 · 측정 2026-06-02 · 선행 감사: platform-audit.html · HONEST-SCOPE.md

원칙 · Principle

"가능한 모든 것을 플랫폼 안에서"
현재 강점(ADK·Gemini·A2A·MCP·Grounding은 이미 라이브)은 유지하고, 나머지를 모두 네이티브 관리형 제품으로 끌어올립니다: 런타임은 Cloud Run → Agent Engine(Agent Runtime), 자체 watchdog/policy/security/compliance/eval/observability는 Govern·Optimize 필러의 관리형 제품으로, 부재(RAG·Search·Sandbox·A2UI·UCP)는 네이티브 도입, 스텁(AP2·connectors)은 공식 SDK·샘플로 완성. 단, gemini-*-flash는 GA라 VAPO 등 region 제품과 호환(D53 유지).

권장 실행 순서 (무제약이라도 의존성은 존재)

키스톤부터 — Agent Engine 위에 나머지가 얹힘
STEP 1 · 키스톤

Agent Engine 배포

Cloud Run→adk deploy agent_engine. 관리형 Sessions·Eval·Observability가 여기서 해금.

STEP 2 · 상태

Sessions + Memory Bank

InMemory 제거 → 관리형 Sessions, VertexAiMemoryBankService 플릿 주입.

STEP 3 · 거버넌스

Identity·Gateway·Registry·Model Armor

IAM 네이티브 ID + Gateway mTLS + Registry(MCP) + Model Armor 전 플릿.

STEP 4 · 최적화

Evaluation·VAPO·Observability

GenAI Eval 22/22 + VAPO 데이터드리븐 + Cloud Trace 네이티브.

STEP 5 · 데이터

RAG Engine·Vertex AI Search

brand-brief RAG corpus + Agent Search 그라운딩.

STEP 6 · 결제

AP2 3-Mandate·UCP·배포

Intent+Cart+Payment 실서명 + Marketplace/Apigee.

① Build / Scale — 런타임·상태 네이티브화 (키스톤)

현재 모든 에이전트가 Cloud Run(FastAPI)에서 돈다. 관리형 Agent Engine으로 옮기면 Sessions·Memory·Eval·Observability가 한꺼번에 네이티브로 붙는다.

컴포넌트네이티브 목표 + 채택 계획공식 레퍼런스 (문서·예제·학습)노력·의존
Agent Runtime현 Cloud Run Cloud Run → Vertex AI Agent Engine(reasoningEngines)
  • adk deploy agent_engine --project --region --display_name로 대표 에이전트 1개부터 배포
  • Cloud Resource Manager + Vertex AI API 활성, ADC
  • vertexai.agent_engines.get(...)로 호출 전환, 22개로 확대
  • Workflow의 HTTP 호출을 reasoningEngines :query로 전환
 ADK deploy(agent_engine) · Use ADK agent · 학습 adk-samples ~1-2주
의존: 런타임 allowlist(D17)
Agent Sessions현 in-proc InMemorySessionService 제거 → 관리형 Sessions
  • Agent Engine 배포 시 AdkApp cloud-managed sessions 자동 사용
  • 세션당 SessionEvents 업로드로 대화이력 영속화
  • cross-invocation 세션 데모(캠페인 재개)로 검증
 Agent Engine Sessions STEP 1에 동반
Agent Memory Bank현 Firestore default Vertex Memory Bank를 플릿 전체에 주입
  • --memory_service_uri=agentengine://<id>로 ADK 연결
  • VertexAiMemoryBankService read/write를 run_agent 도입/종료부에 배선(P3.2)
  • memory topics 구성 + similarity 검색으로 프롬프트 주입
  • memory poisoning 방어 = Model Armor 인스펙션(③)
 Memory Bank overview · set-up · ADK Memory 6-8hr 코어
+ 22-agent 회귀
Agent Sandbox현 부재 코드실행 에이전트용 Agent Sandbox(gVisor) 도입
  • analyst의 사용자-공급 분석쿼리를 Sandbox에서 실행
  • GKE Autopilot + gVisor 풀(D23) 연결
 Agent Engine(Sandbox) 2-3일 (코드실행 도입 시)
Agent Studio / Garden현 설계용 신규 에이전트 시각 프로토타이핑 + Garden 시드 채택
  • Agent Studio에서 새 Tier-1 에이전트 프로토타입 → 코드락
  • Agent Garden 샘플(고객서비스·RAG 등)을 시드로 import
 Agent Builder · 예제 agent-starter-pack / adk-samples 평가 1주

② Govern — 자체구현 → 관리형 필러 (가장 큰 갭)

Identity·Gateway·Registry는 플랫폼 Govern 필러의 핵심. 현재 watchdog/policy/security/compliance는 전부 우리 에이전트 → 관리형 거버넌스로 전환하면 "제품 사용"을 정직하게 주장 가능.

컴포넌트네이티브 목표 + 채택 계획공식 레퍼런스노력·의존
Agent Identity현 SPIFFE 선언 플랫폼 네이티브 IAM principal로 승격
  • 각 에이전트를 first-class IAM principal(고유 암호 ID)로 등록
  • least-privilege 정책·resource boundary 부여
  • A2A 전송계층에서 caller identity 검증
 Govern: Identity 1주 · IAM
Agent Gateway현 mTLS 미집행 Gateway에서 mTLS 집행 + MCP 등록
  • A2A ingress 앞단에 Agent Gateway 배치(트래픽 "관제")
  • mTLS enforcement + Model Armor 정책 인라인
  • tiktok-mcp-server를 Gateway에 MCP로 등록
 Govern: Gateway 1-2주 · Private Preview 승인
Agent Registry현 자체 list 자체 registry → 플랫폼 Agent Registry
  • 22 에이전트 + MCP 서버를 네이티브 Registry에 등록(이미 GE Agent Gallery 등록은 입증)
  • Gateway·Identity·Policy와 동일 거버넌스로 노출
 Govern: Registry 3-5일
Model Armor현 A2A만 전 플릿 + Memory Bank 인라인
  • input/output 템플릿 분리(주입·jailbreak·PII·악성URL)
  • 22-agent generateContent 경로 + Memory Bank에 인라인 적용(B4)
  • Codelab 패턴으로 차단 검증
 Model Armor docs · Vertex 통합 · 학습 Codelab 1-2일
Agent Policy / Security / Compliance현 자체 3종 자체 watchdog → 네이티브 Govern 제품
  • 정책 게이트 7종을 Agent Policy로 표현
  • security_watch/anomaly_watch → Agent Security + Agent Anomaly Detection 관리형으로 대체 평가
  • compliance → Agent Compliance + ADK Safety 가이드
 ADK Safety · Govern 필러 각 1주 · 제품 가용성

③ Optimize — 평가·최적화·관측 네이티브화

현재 eval 2/22(자체 골든) + 로컬 결정론 triage 패스 + OTel(export 게이트). 전부 플랫폼 Gen AI Eval + VAPO + Cloud Trace로 승격.

컴포넌트네이티브 목표 + 채택 계획공식 레퍼런스노력·의존
Agent Evaluation현 2/22 자체 Gen AI Evaluation service로 22/22
  • Agent Engine 배포 후 GenAI Client로 평가 실행
  • 루브릭 메트릭(FINAL_RESPONSE_QUALITY·TOOL_USE_QUALITY·HALLUCINATION·SAFETY) + 궤적(trajectory_in_order_match)
  • 22 에이전트별 eval dataset + CI 게이트, holdout 70% floor(D54)
 Evaluate(GenAI Client) · 소개 블로그 ~10인일 · STEP1 선행
Agent Optimizer현 로컬 패스 로컬 triage → Vertex AI Prompt Optimizer(VAPO) 데이터드리븐
  • client.prompt_optimizer.optimize(method="vapo", config=...) 실행 데이터로 시스템 instruction 개선
  • prompt template + 데이터셋(triage 56케이스)으로 GA 모델 대상 튜닝
  • before/after(40.5→100% train·71.4% holdout) 재현 + 라이브 1회
 VAPO docs · 예제 notebook 2-3일 · GCS 버킷·ADC
Agent Observability현 OTel export 게이트 Cloud Trace 네이티브 export 상시화
  • SS_OTEL_ENABLED 상시 ON + Cloud Trace/Chronicle export
  • per-agent cost attribution 대시보드(P3.3)
  • agent.latency_ms warm-path SLO 측정
 Agent Engine Observability 2-3일
Agent Simulation현 자체 triage_sim 자체 시뮬 → 플랫폼 Agent Simulation
  • nightly Agent Simulation을 관리형 제품으로 이관 평가
  • holdout regression을 시뮬에 연결(P3.4)
 Optimize 필러 1주 · 제품 가용성

"라이브 6 → 거의 전부 네이티브 라이브"
