System Gap Analysis · 자율 완주 vs 최소 HITL

우리 시스템은 "우리리우 캠페인"처럼 끝까지 자율 완주할 수 있는가

사람이 6단계를 수동 완주한 실제 캠페인(우리리우 2차 발송)을 기준선으로, v2 에이전트 시스템(TS 제품 스택 + Python ADK 챌린지 스택 통합)이 각 단계를 어디까지 자동화했는지, 그리고 진짜 사람이 필요한 지점에만 HITL을 남기려면 무엇이 빠졌는지를 코드 감사로 분석합니다.

핵심 결론 (먼저)

① 데이터 모델은 이미 정렬됨 — v2의 CampaignStage 6단계(overview→sourcing→outreach→shipping→content_review→performance)가 우리리우 캠페인의 6 workflowSteps와 정확히 1:1 일치. 설계 의도는 맞다.

② 역설적 강점 — 우리리우는 마지막 성과분석 단계에서 멈췄다(pending, KPI 전부 0). v2는 바로 그 단계를 campaign-progression이 자동 완료한다. 사람이 못 끝낸 곳을 시스템이 닫는다.

③ 진짜 갭은 3가지 — (A) 조립 갭: 메인 brand-campaign 워크플로가 outreach 단계에서 종료, 3~6단계는 별도 이벤트 워크플로로 흩어져 "한 흐름 자율 완주"가 아직 아님. (B) HITL 배치 갭: 게이트 5종이 전부 기본 always_ask(과잉), 게다가 사용자가 필수로 본 콘텐츠 승인·예산/계약 게이트는 존재하지 않음. (C) 라이브 갭: 실제 인플루언서 발송·배송·성과는 아직 stub/operator-deploy 단계.

03 · 갭 A — 조립(Composition) 갭

GAP-A · 가장 큰 구조적 갭

brand-campaign 워크플로가 outreach 단계에서 종료된다

메인 packages/workflows/src/workflows/brand-campaign.ts는 Stage 1(overview) + Stage 2(sourcing+vetting+approveShortlist)만 실행하고 stage="outreach"로 핸드오프하며 끝납니다. 코드 주석도 명시: "Phase 2+ adds outreach / shipping / content_review / performance". 즉 3~6단계는 별도 이벤트 워크플로(creator-track·shipment-tracking-poller·tiktok-post-poller·report-deliver·campaign-progression)로 흩어져 있고, "하나의 캠페인을 끝까지 자율로 미는" 단일 흐름으로 조립돼 있지 않습니다.

영향 — 우리리우처럼 "브리프 1개 → 성과리포트까지" 연속 완주를 시연하려면, 흩어진 이벤트 워크플로를 단계 전이(approveStageAdvance)로 잇는 오케스트레이션 조립이 필요합니다. 부품은 다 있으나 파이프가 끊겨 있습니다.

04 · 갭 B — HITL 배치 갭 (사용자 핵심 요구)

현재 정책 게이트는 5종이고 기본정책이 전부 always_ask(packages/capabilities/src/workspace/policy.ts 기본값)입니다. 사용자가 승인한 "필수 HITL = {배송, 콘텐츠 최종 승인, 예산/계약}" 기준으로 보면 두 종류의 갭이 있습니다.

GAP-B1 · 과잉 HITL (마찰)

자동화해야 할 4개 게이트가 기본 always_ask로 사람을 부른다

사용자는 아웃리치 발송·응답을 자동화 대상으로 분류했습니다. 그런데 현재는:

게이트현재 기본목표

approveOutreachSendalways_askauto_unless (스팸점수·대량발송 시만)
approveReplyResponsealways_askauto_unless (부정·이탈·민감어 시만)

게이트 엔진은 auto_unless 모드(술어 매칭 시에만 사람 호출)를 이미 지원합니다 — 코드 변경이 아니라 기본 정책값 + 술어 정의가 갭입니다.

GAP-B2 · 필수 게이트 누락

사용자가 필수로 본 게이트 2개가 존재하지 않는다

GateKind/GateConfig에 approveContent와 approveBudget(계약 집행)를 신설해야 사용자의 "최소 HITL" 정의가 완성됩니다.

07 · 권고 — 우선순위

P1 · HITL 정책 재배치 (가장 작은 변경, 가장 큰 효과) — 기본 정책값을 사용자 기준으로 교체: 4개 게이트 auto_unless/auto화 + 위험 술어 정의. 코드가 아니라 정책 기본값 + 술어. 저난도

P2 · 누락 게이트 2종 신설 — GateKind/GateConfig에 approveContent·approveBudget 추가, content_review·예산 집행 지점에 배선. 중난도

P3 · 조립 갭 해소 — brand-campaign을 Stage 3~6까지 확장하거나, 흩어진 이벤트 워크플로를 approveStageAdvance 전이로 잇는 단일 자율 흐름 구성. 고난도·핵심

P4 · 라이브화 — 실 캠페인 1건으로 outreach→shipment→content→performance를 운영자 ADC로 실행, Imagen W7 stub→live 승격. 운영자 의존

분석의 한계 (정직) — 자동화 상태는 코드 감사(파일·게이트·HONEST-SCOPE) 기반입니다. "부품 존재·미조립" 판정은 brand-campaign이 outreach에서 종료한다는 코드 근거에 따른 것으로, 별도 이벤트 워크플로가 각 단계를 부분적으로 처리합니다 — 즉 "완전 미구현"이 아니라 "단일 흐름 미조립"입니다. P1~P4는 제안이며, 실제 착수 전 단계별 설계 검토가 필요합니다.
